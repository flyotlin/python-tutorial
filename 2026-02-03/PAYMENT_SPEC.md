# 支付處理規格 (Payment Processing Spec)

## 1. 目標

實作一個「支付處理」功能：給定支付方式、金額，以及該方式所需的參數，執行對應的付款流程並輸出結果。

## 2. 支付方式與參數

| 支付方式       | ID          | 必要參數                         | 說明           |
|----------------|-----------------|----------------------------------|----------------|
| 信用卡         | `credit_card`   | `card_no`, `expiry`              | 卡號、到期日   |
| 加密貨幣       | `crypto`        | `wallet_addr`, `network`         | 錢包地址、鏈   |

- 金額 `amount` 為數字，所有方式皆需要。
- 非該方式用到的參數可為 `None` 或省略。

## 3. 行為規格

### 3.1 共同流程

- 印出 `--- 處理金額: {amount} ---`。
- 依「支付方式」執行對應邏輯（見下表）。
- 若該方式必要參數有缺或為空，則 **raise ValueError**，訊息需明確（例如「信用卡資訊不完整」「加密錢包資訊不完整」）。

### 3.2 信用卡 (`credit_card`)

1. 檢查 `card_no`、`expiry` 皆存在且非空。
2. 印出：`驗證卡片 {card_no}，到期日 {expiry}...`
3. 印出：`刷卡成功！`

### 3.3 加密貨幣 (`crypto`)

1. 檢查 `wallet_addr`、`network` 皆存在且非空。
2. 印出：`連線至 {network} 網路...`
3. 印出：`發送交易至 {wallet_addr}...`
4. 印出：`轉帳發送成功！`

## 4. 介面（給學生思考用）

- **輸入**：支付方式、金額、以及該方式所需的參數（可選或必填依上表）。
- **輸出**：無回傳值；以 `print` 輸出流程訊息；錯誤時以 `ValueError` 傳遞。

請先思考：

1. 若用「一個函數」實作所有方式，函數簽名會長怎樣？參數會如何變多？
2. 若未來要加「第三種支付方式」，你要改哪裡、加多少 `elif`？
3. 有沒有辦法讓「新增一種支付方式」時，只加新程式、少改舊程式？

寫完一版後，可對照 `demo_payment_bad.py`（單一函數版）與 `demo_payment_good.py`（類別／多型版），比較哪一種比較符合上面第 2、3 點的考量。
